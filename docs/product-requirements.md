# Meeting Copilot 产品需求文档

> 日期：2026-06-18  
> 阶段：需求设计 / 可行性验证  
> 产品定位：中文技术会议实时 AI Copilot，不是通用音频转文字工具。

## 1. 一句话定义

Meeting Copilot 面向中文技术团队，在会议中实时识别议题、候选决策、行动项、风险和未闭环问题，以低打扰建议卡片提醒关键缺口，并在会后输出带证据时间戳的工程纪要。

它的价值不在于“把音频变成文字”，而在于：

- 会中发现工程讨论缺口，例如 owner、deadline、rollback、test、monitoring 没有闭环。
- 把会议讨论持续维护成结构化状态，而不是会后从一堆文字里重新整理。
- 所有建议、决策、待办都必须带证据片段，避免 AI 空口断言。
- 针对中文技术会议优化术语、字段、服务名、指标、错误码和中英混排。

## 2. 目标用户

### 技术负责人 / 架构师

痛点：

- 会议中要同时听方案、判断风险、追问边界，容易漏掉关键工程约束。
- 会后需要把讨论变成决策、风险和后续行动。

Meeting Copilot 应提供：

- 架构风险、兼容性、灰度、回滚、监控、依赖方提醒。
- 候选决策与被推翻决策记录。
- 可复盘证据片段。

AI 不能做：

- 不能替技术负责人判断方案一定可行或不可行。
- 不能把候选决策当成正式决策。

### 会议主持人 / 项目负责人

痛点：

- 会议容易跑偏，行动项缺 owner 或 deadline。
- 会议结束后才发现很多问题没有闭环。

Meeting Copilot 应提供：

- 议题切换和未闭环问题提示。
- owner/deadline/验收标准缺失提醒。
- 会后结构化纪要和待确认清单。

AI 不能做：

- 不能自动分配责任人。
- 不能未经确认创建正式 issue。

### 一线工程师 / SRE

痛点：

- 会中信息量大，容易错过自己负责的事项。
- 事故、上线、接口评审里的技术细节容易在纪要中丢失。

Meeting Copilot 应提供：

- 与自己相关的行动项、风险和时间点。
- API、字段、服务名、指标等技术实体高亮。
- 会后可检索的证据链。

AI 不能做：

- 不能把不明确的 owner 强行归给某个人。
- 不能用会议内容做人事、绩效或责任裁定。

## 3. MVP 目标

MVP 只证明一个核心命题：

> 在中文技术会议中，实时 Copilot 能比普通转写/总结工具更早发现工程缺口，并输出可信的结构化证据。

MVP 必须做：

- 桌面端手动开始、暂停、停止会议。
- 麦克风与系统音频采集，至少区分本机输入和系统混音。
- 实时 ASR 接入，默认优先本地/开源 ASR，中文技术会议质量先通过 bake-off。
- partial/final/revision 转写稳定器。
- 项目级技术词库和热词入口。
- 会议状态机：议题、候选决策、行动项、风险、未闭环问题。
- 工程语境门禁：非软件工程会议不触发工程缺口卡片。
- 状态事件日志：记录状态 created/updated/answered/confirmed/dismissed。
- 缺口雷达：owner、deadline、rollback、test/verification、metric/monitoring。
- 建议卡片：低频、带证据、可保留、可忽略、可标记错误。
- 会后结构化纪要：背景、决策、待办、风险、未闭环问题、证据时间戳。
- Markdown/JSON 导出。
- 本地会话管理与删除。
- 隐私控制：手动开启、状态可见、可暂停、可删除。

MVP 明确不做：

- 不做隐蔽录音或全天监听。
- 不自动加入所有日历会议。
- 不默认对所有参会人广播 AI 建议。
- 不做完整多人身份识别；若 owner 不明确，标记为待确认。
- 不自动创建 issue 或自动变更项目状态。
- 不替用户做架构、上线、法律、人事或合规裁决。
- 不把低置信度 ASR 文本用于强建议。
- 不承诺所有会议软件和所有声卡场景首版兼容。
- 不把 Whisper 或任意单一模型当作未经评测的默认中文主链路。
- 不把远程 ASR 作为默认必选链路。
- 不新增隐藏收费项；任何远程 ASR 或付费 provider 都必须用户显式配置。
- 不做通用会议问答或泛化 AI 助手。
- 不做全量知识库检索。
- 不做自动周报或跨会议长期记忆。
- 不做完整协作空间。
- 不把多人 speaker diarization 作为 MVP 强依赖。

## 4. 核心用户旅程

### 会前

1. 用户选择开始一次会议。
2. 客户端检查麦克风、系统音频权限和电平。
3. 用户选择项目词库或导入临时热词。
4. 客户端展示录音/转写/云端调用状态。
5. 用户可复制会议告知文案，用于提醒参会者。

验收点：

- 用户能在 30 秒内确认音频输入正常。
- 权限或设备异常必须明确提示，不允许静默失败。
- 用户能看到哪些数据会发送给 ASR/LLM provider。

### 会中

1. 客户端实时接收 mic/system audio。
2. ASR 输出 partial/final/revision。
3. 转写稳定器生成可用于状态机的稳定片段。
4. 系统先判断是否处于软件工程/研发协作/上线交付语境。
5. 会议状态机增量更新当前议题、候选决策、待办、风险、未闭环问题。
6. 每次状态变化写入状态事件日志。
7. 缺口雷达只在有足够证据和仍可追问时生成低频建议卡片。
8. 用户可以保留、忽略、复制追问、标记误报。

验收点：

- 建议卡片每场会议默认 3-8 条，不刷屏。
- 每张建议卡片必须引用证据时间戳和原始文本片段。
- 建议卡片应在相关 final segment 后 10-30 秒内出现；太晚只能降级为会后待确认项。
- 主视图以会议状态和建议卡片为中心，transcript 作为可展开证据面板。
- 非工程会议可以生成摘要，但工程建议卡片必须为 0。
- 低置信度 ASR、嘈杂音频、LLM 超时必须降级，不输出强结论。

### 会后

1. 客户端保存原始音频、raw transcript、normalized transcript 和结构化状态。
2. LLM 生成带证据的会议纪要。
3. 用户检查待确认项，例如 owner、deadline、候选决策。
4. 用户导出 Markdown/JSON。
5. 用户可以删除会议，删除应覆盖音频、转写、中间结果和导出物。

验收点：

- 纪要中的每条决策、行动项、风险都能跳回证据片段。
- 未确认内容必须标记为“待确认”，不能混入正式结论。
- 删除语义可解释、可执行。

## 5. 功能需求

### 音频采集

- 支持手动开始、暂停、恢复、停止。
- 支持麦克风轨、系统音频轨、混合轨。
- 记录采样率、声道数、设备名称、采集异常。
- 监控电平、静音、丢帧、设备切换。
- 任何音频质量问题都要进入会议质量日志。

### 实时 ASR

- ASR provider 必须通过中文技术会议 bake-off 后才可作为主链路。
- MVP 默认优先本地 ASR，例如 FunASR streaming 或 sherpa-onnx。
- 远程 ASR 只作为评测对照、可选高质量模式或企业自选 provider。
- 保留 raw transcript、normalized transcript、revision 历史。
- 支持项目级热词。
- 支持中英混排、数字、百分比、错误码、字段名、服务名。
- 真实 provider 报告必须记录失败、超时、空转写、不可评测样本。

### 转写稳定器

- partial 文本只能用于低风险预览和状态候选。
- final 文本才能进入稳定证据。
- revision 必须能回滚或修正先前片段。
- LLM 只能基于稳定片段生成强建议。

### 会议状态机

- 维护 Topic、DecisionCandidate、ActionItem、Risk、OpenQuestion。
- 每个状态对象必须引用 EvidenceSpan。
- 状态有置信度和确认状态。
- 状态更新必须可追踪：由哪段 transcript 触发，是否被用户确认或忽略。
- 每次状态创建、更新、合并、回答、确认、忽略都必须写入 MeetingStateEvent。

### 缺口雷达

MVP 只做以下缺口：

- owner 缺失。
- deadline 缺失。
- rollback 缺失。
- test/verification 缺失。
- metric/monitoring 缺失。

触发原则：

- 只有 `meeting_context.is_engineering_meeting = true` 才触发工程缺口卡片。
- 只在检测到候选决策、行动项、上线意图、风险承诺后触发。
- 头脑风暴、背景介绍、闲聊阶段降低触发。
- 同类建议需要冷却时间。
- 置信度不足时进入“待观察”，不打扰用户。
- 如果已错过追问窗口，降级为会后待确认项。

### 建议卡片

每张卡片包含：

- 类型。
- 建议追问。
- 证据片段。
- 时间戳。
- 置信度。
- 触发原因。
- 用户操作：保留、忽略、复制、标记错误、转为待确认项。

建议语气：

- 使用“可能缺少”“建议确认”“是否需要追问”。
- 不使用“必须”“一定”“不可上线”等强裁判式表达。

### 会后纪要

Markdown 输出至少包含：

- 会议基本信息。
- 背景和议题。
- 已确认决策。
- 候选/待确认决策。
- 行动项。
- 风险和缓解方案。
- 未闭环问题。
- 技术实体清单。
- 证据引用。

JSON 输出必须保持结构化，便于后续接入 Jira、Linear、GitHub、MCP 或内部系统。

## 6. 不可违反验收标准

- 默认平台：MVP 首发 macOS Apple Silicon；Windows 不作为第一轮阻塞目标。
- 默认成本：默认 ASR 链路不得产生远程 API 费用；远程 ASR 必须默认关闭。
- LLM 成本：每场会议必须记录 LLM provider、model、prompt version、调用次数、失败/重试次数、token 或等价用量。
- 实时性：建议卡片必须在相关 final segment 后 10-30 秒内出现；超窗只能计为会后待确认项。
- EvidenceSpan：DecisionCandidate、ActionItem、Risk、OpenQuestion、SuggestionCard、MeetingSummary 正式内容必须引用 EvidenceSpan。
- 事件日志：任何状态创建、更新、合并、回答、确认、忽略都必须写 MeetingStateEvent。
- 工程语境门禁：非软件工程会议工程缺口卡片必须为 0。
- 工程上下文 gate：没有候选决策、行动项、上线/变更/风险信号时，不得触发工程缺口卡片。
- 项目事实边界：没有用户显式导入 repo/docs/issue 时，不得声称基于代码库事实判断。
- ASR 质量：raw ASR 指标和 normalized 指标必须同时记录，不能只展示修正后结果。
- 建议安全：不得输出“必须上线/不可上线/方案错误”等裁判式结论，不做人事或责任裁定。
- 降级：ASR 低质量、LLM 超时、JSON schema 失败、证据链不完整时，必须沉默、待确认或会后降级。
- 删除：删除会议必须覆盖音频、临时 chunk、raw transcript、normalized transcript、LLM 中间输出、状态对象、建议卡片、导出文件和本地缓存。
- 密钥：API key 不得进入 manifest、报告、日志、导出物或测试 fixture。

## 7. 成本预算与调用策略

默认预算目标：

- 默认不对每个 ASR partial 调 LLM。
- 会中增量分析默认每 20-45 秒或关键状态变化后触发。
- 同一议题内同类缺口卡片默认 5 分钟冷却。
- 一场 60 分钟会议的会中 LLM 调用目标不超过 80 次；更优目标是 20-40 次。
- 会后报告允许一次较大上下文调用，但必须记录 token 或等价用量。
- LLM 失败重试默认最多 1 次；连续失败进入本地记录和会后分析降级。

超预算策略：

- 降低会中分析频率。
- 只保留状态抽取，不生成新建议卡片。
- 转为会后待确认项。
- 明确在质量状态中显示“LLM 调用已降级”。

## 8. 成功指标

ASR 指标：

- 中文 CER <= 12%。
- 核心技术实体 recall >= 90%。
- 核心技术实体 precision >= 90%。
- 热词后核心术语准确率 >= 90%，并且相对无热词有明确提升。
- partial latency P50 <= 1.5s，P95 <= 2.5s。
- final latency P95 <= 3.5s。

Copilot 指标：

- 每场有效建议卡片 3-8 条。
- 建议卡片保留率 >= 40%。
- 建议卡片误报率 <= 20%。
- 决策抽取 precision/recall >= 85%。
- 行动项 owner/deadline 提取准确率 >= 80%；不确定时允许标记待确认。
- 证据引用准确率 >= 95%。
- 幻觉率接近 0；无证据内容不得进入正式纪要。
- 会后纪要人工修改率逐步下降。

体验指标：

- 会议开始前音频检查成功率 >= 95%。
- ASR/LLM provider 单点失败不导致整场会议数据丢失。
- 用户能明确看到录音和云端调用状态。

## 9. Go / No-Go

Go：

- 中文 ASR bake-off 达到 MVP 门槛。
- 本地/开源 ASR 至少能在目标硬件上稳定输出实时中文文字；否则必须重新评估产品成本结构。
- 至少两个真实中文技术会议场景中，建议卡片被用户认为有用。
- 会后纪要能稳定产出带证据的决策、待办、风险和未闭环问题。
- 低置信度时系统能沉默或降级，而不是编造。

No-Go：

- ASR 无法稳定识别中文技术实体。
- 建议卡片误报过多，干扰会议节奏。
- 纪要无法追溯证据。
- 低置信度文本被 LLM 放大成错误建议。
- 隐私状态不透明，用户无法控制录音和上传。
