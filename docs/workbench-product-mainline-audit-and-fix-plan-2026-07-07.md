# PC Web MVP 产品主线验收与修复计划 - 2026-07-07

> 状态：执行中
> 目标：把当前工作台从“工程验证台”收敛成能跑主流程的 PC Web MVP：开始会议/导入录音 -> 实时文字 -> 实时建议 -> 会后复盘 -> 历史记录/删除。
> 方法：多 Agent 只读审查 + 本地交叉验证 + TDD 修复 + 浏览器逐按钮自测。
> 边界：本轮不做桌面安装包、系统音频采集、移动端、远程 ASR 接入、真实生产发布；先把 Web MVP 主链路补齐。

## 1. 多 Agent 审查结论

### Agent A：前端按钮与功能

当前按钮都有后端路径，但页面状态一致性不足。

| 控件 | 当前能力 | 主要问题 | 优先级 |
|---|---|---|---|
| 加载样本会议 | 创建 mock ASR session，渲染转写和候选建议 | 文案偏 Demo；新会话不会统一清理旧状态 | P1 |
| 上传音频文件 | 上传音频 -> batch ASR -> session -> 事件渲染 | 失败提示存在 HTML 注入风险；重复上传状态可能覆盖 | P1 |
| 生成建议卡片 | 手动触发 LLM 建议卡片 | 不是自动建议；重复点击会重复追加；结果未持久化 | P0 |
| 方案考量 | 手动触发方案卡 | 文案抽象；重复点击重复追加；结果未持久化 | P1 |
| 真实录音 | 麦克风 -> WebSocket -> ASR -> 停止后落库 | 按钮不显示“结束会议”；录音中/整理中状态不清楚 | P0 |
| 实时订阅 | SSE 回放已落库事件 | 文案是技术词；`transcript_partial` 渲染不一致 | P1 |
| 删除会话 | 删除当前 ASR live session | 前端计数/底部状态/currentEvents 不完全清理；录音中删除未停麦克风 | P0 |

### Agent B：产品需求覆盖

当前状态不是 0，也不是生产可用。

```text
Local Shadow Preview: 已成立
真实麦克风普通中文 ASR 链路: 已初步成立
真实中文技术会议 Copilot: 尚未成立
Production MVP: 未成立
```

已做得比较扎实的是 EvidenceSpan、工程语境门禁、模拟/回放链路、真实麦克风到本地 ASR 的物理链路。未满足原始产品承诺的是：真实中文技术会议中的实时工程建议、系统音频、原始录音保存、正式会后纪要、历史会话产品入口、用户反馈指标。

### Agent C：UI/UX 简化

当前页面是“开发调试工作台”，不是会议用户界面。P0 要把首屏改成：

```text
开始会议 / 结束会议
导入录音
实时文字
实时建议
会后复盘
历史记录
```

首屏应移除或折叠这些术语：ASR、LLM、events.sse、sidecar、gateway、confidence、evidence、tokens、model、candidate、rule_id、会话落库、级联清除、归一化。

### Agent D：后端/API 能力

后端已具备骨架：

```text
录音/上传音频
  -> ASR live session
  -> events / candidates
  -> 手动触发 LLM 建议卡片 / 方案卡 / 纪要
```

但缺产品后端能力：

- 缺 `GET /live/asr/sessions` 会话列表。
- LLM 建议卡、方案卡、纪要没有写回 live session，刷新后不可作为历史结果读取。
- 缺已生成纪要的下载端点。
- 缺原始录音音频保存。
- SSE 目前更像已落库事件回放，不是真正持续 pub/sub。

## 2. 本轮范围判断

### 本轮必须修

- 页面主流程文案和信息架构：让用户知道下一步该点什么。
- 录音按钮状态：未开始是“开始会议”，录音中是“结束会议”，整理中是“正在整理...”。
- 删除会话完整清理前端状态。
- 同时支持 WebSocket 原始 `partial/final` 与 SSE `transcript_partial/transcript_final` 渲染。
- 会后复盘入口接入现有 `/minutes`。
- 历史记录入口接入新增会话列表 API。
- 建议卡、方案卡、纪要最小持久化到 live ASR session。
- 浏览器逐按钮自测记录。

### 本轮暂不做

- 系统音频采集。
- Tauri/Mac/Windows 安装包。
- 原始音频 WAV/PCM 保存。
- 录音中跨页面恢复。
- 远程 ASR provider。
- 长会生产级稳定性和 AudioWorklet 迁移。

这些不是不重要，而是会把本轮重新拖回大工程循环。当前先让 Web MVP 的产品主流程成立。

## 4. 验收标准

本轮完成后，PC Web MVP 至少应满足：

1. 用户打开页面能看到清晰主路径：开始会议、导入录音、实时文字、实时建议、会后复盘、历史记录。
2. 示例会议可一键加载，并显示文字、建议、会后复盘入口和历史记录。
3. 真实录音按钮状态清楚，第二次点击就是结束会议。
4. 删除本次会议后，页面不留下旧计数、旧卡片状态或旧会话元信息。
5. 生成会议建议、方案分析、会后复盘后，刷新/历史读取不丢核心结果。
6. 后端有会话列表 API，前端可展示最近会议。
7. 所有本轮变更有自动化测试和浏览器自测记录。

## 5. 产品价值判断

本轮不是为了宣布 Production MVP ready，而是为了让产品主线不再散：

```text
当前目标：PC Web MVP mainline usable
不是目标：Production release / 桌面安装包 / 系统音频完整采集
```

后续真正决定产品是否值得继续投入的验收仍然是：

```text
真实授权中文技术会议
  -> 本地实时 ASR
  -> 10-30 秒内出现 1-3 张有证据、有帮助的工程提醒
  -> 会后生成可追溯纪要
```

如果真实技术会议里只能得到转写，不能得到及时、有证据、低打扰的提醒，这个产品就不成立。
