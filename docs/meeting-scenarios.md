# 中文技术会议场景模板

> 日期：2026-06-18  
> 目的：把“中文技术会议”拆成可评测、可设计、可验收的场景，避免产品泛化成普通会议助手。

## 1. 场景设计原则

每类会议都要定义：

- 典型用户。
- 关键技术实体。
- 典型决策。
- 典型风险。
- 常见缺口。
- 实时建议卡片。
- 会后输出结构。
- ASR/LLM 评测样本。

MVP 优先打穿两个场景：

- API 评审会。
- 上线 / 灰度评审会。

事故复盘和架构设计作为第二批高价值场景。

## 2. API 评审会

目标：

- 确认接口字段、兼容性、错误码、鉴权、调用方影响。

关键实体：

- endpoint、HTTP method、字段名、枚举值、错误码、版本号。
- trace_id、request_id、tenant_id、user_id 等字段。
- 调用方、下游服务、网关、鉴权、限流。

典型决策：

- 是否新增字段。
- 是否兼容老版本。
- 是否需要灰度。
- 错误码和重试策略。

常见缺口：

- 没有明确调用方影响。
- 没有兼容方案。
- 没有测试用例。
- 没有回滚策略。
- 没有 owner 或 deadline。

建议卡片示例：

```text
类型：compatibility_gap
建议：刚才形成了新增 trace_id 字段的候选决策，但还没有听到老版本调用方兼容方案。是否需要确认默认值、灰度范围和回滚方式？
证据：12:31-12:48 “接口新增 trace_id 字段……”
```

会后输出：

- API 变更摘要。
- 字段与错误码变更。
- 兼容性结论。
- 调用方影响。
- 测试清单。
- 待确认问题。

## 3. 上线 / 灰度评审会

目标：

- 确认上线范围、灰度比例、监控指标、回滚阈值和值班责任人。

关键实体：

- 灰度比例、发布窗口、服务名、版本号、监控指标。
- QPS、P95/P99、错误率、CPU、内存、Kafka lag。
- 回滚、熔断、开关、feature flag。

典型决策：

- 是否上线。
- 灰度比例。
- 监控指标和阈值。
- 回滚负责人。

常见缺口：

- 没有 rollback owner。
- 没有明确指标阈值。
- 没有验证计划。
- 没有灰度停止条件。
- 没有通知依赖方。

建议卡片示例：

```text
类型：rollback_gap
建议：上线方案已讨论到灰度百分之十，但还没有明确回滚负责人。是否需要确认错误率超过多少由谁执行回滚？
证据：18:10-18:29 “先灰度百分之十……”
```

会后输出：

- 上线窗口。
- 灰度计划。
- 监控指标。
- 回滚条件。
- 值班和 owner。
- 风险与缓解方案。

## 4. 事故复盘会

目标：

- 还原时间线、影响范围、根因、缓解动作和预防措施。

关键实体：

- 告警时间、恢复时间、影响用户、SLA/SLO。
- 错误码、日志字段、trace_id、服务名、数据库、队列。
- CPU、内存、P99、QPS、超时、重试。

典型决策：

- 根因判断。
- 短期修复。
- 长期预防。
- 监控补充。

常见缺口：

- 时间线不完整。
- 根因和诱因混淆。
- 预防措施没有 owner。
- 没有验证防复发方式。
- 没有更新告警策略。

建议卡片示例：

```text
类型：verification_gap
建议：当前已经提出限流修复，但还没有听到如何验证不会再次触发同类告警。是否需要补充压测或回归指标？
证据：25:40-26:12 “先把限流阈值调低……”
```

## 5. 架构设计会

目标：

- 讨论系统边界、依赖、扩展性、可靠性和演进成本。

关键实体：

- 服务名、模块名、数据库、缓存、消息队列、网关。
- K8s、Redis、MySQL、Kafka、RocketMQ、ES。
- 一致性、幂等、降级、熔断、容灾。

典型决策：

- 选型方案。
- 数据模型。
- 同步/异步边界。
- 扩容和降级策略。

常见缺口：

- 没有容量估算。
- 没有故障模式。
- 没有数据一致性策略。
- 没有迁移方案。
- 没有兼容老链路。

## 6. 需求澄清会

目标：

- 把业务目标转成技术边界和可验收需求。

关键实体：

- 用户角色、业务流程、状态流、异常路径。
- 指标、验收标准、上线时间。

常见缺口：

- 验收标准不清。
- 边界条件不清。
- 数据来源不清。
- 依赖方未确认。

## 7. 排期 / 拆任务会

目标：

- 将工作拆成可执行任务，确认 owner、deadline、依赖和验收。

关键实体：

- 任务、owner、deadline、依赖、风险、里程碑。

常见缺口：

- 任务没有 owner。
- deadline 不明确。
- 依赖没有责任人。
- 验收标准缺失。
