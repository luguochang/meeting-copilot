# Meeting Copilot 项目诊断与优化方案

> 日期：2026-07-06  
> 性质：第三方代码与执行复盘 + 优化路线图  
> 范围：`code/`、`docs/`、`tools/`、`tests/`、`configs/` 全量检索 + 关键代码精读

---

## 0. 一句话结论

**产品方向是对的，方法学错了。** 项目用 17 天产出了 157 个文档、220 条决策记录、87 个 pcweb 计划，却从未让一条真实音频走完"采集→ASR→状态→LLM→卡片→纪要"的主链路。根因不是技术能力不足，而是陷入了"用规划/审批/就绪门/证据脚手架的数量冒充产品进展"的循环——这一点 7 月 5 日的 reset 文档自己已经承认。真正的主链路代码约 1.66 万行（占总非测试代码 35%），其余 65% 是 `approval/readiness/preflight/skeleton/boundary/no-op` 脚手架。前端把这套 no-op 状态机直接渲染成了 UI，所以"丑、看不懂、交互差"。后端 LLM 全程禁用、音频采集全 no-op、零日志、单文件 8878 行，所以"不可用、不稳定、不生产级"。

## 3. 代码现状诊断

### 3.1 后端：能启动，但主链路是断的

**能跑的部分**：FastAPI `create_app()` 可启动，`GET /health` → 200，`GET /docs` Swagger 可用，25 个真实业务端点（会话 CRUD、SSE 事件流、report.md、suggestion-candidates 等），471 测试通过。core 模块（`contracts.py`/`gates.py`/`session_snapshot.py`）质量最高——EvidenceSpan/SuggestionCard 数据模型、gate 验证、Markdown 报告生成都是真实可用的。

**断的部分**：
- **LLM 完全禁用**。`app.py:995-1004` 对 LLM 执行端点显式 `raise HTTPException(422)` 拒绝非 disabled 模式；`app.py:1855-1862` 执行结果恒为 `run_status="skipped"`。整个 web backend 没有 `httpx`/`requests`/`openai` 依赖，**从不发起任何 LLM HTTP 调用**。讽刺的是 `code/asr_runtime/scripts/meeting_analysis.py:105-151` 有真实可用的 `call_llm_with_usage()`（urllib POST 到 `/v1/chat/completions`），但从未被 web backend 调用——**两条代码路径从未合并**。
- **音频采集全 no-op**。`asr_worker_sidecar.py` 头注释明写"intentionally avoids process spawning, audio capture, model imports"；18 个 `FALSE_SAFETY_FLAGS` 全 false。
- **卡片永不创建**。`card_status` 恒为 `not_created`。

**生产级规范缺陷**：

| 缺陷 | 严重度 | 证据 |
|---|---|---|
| 零结构化日志 | 严重 | `grep -rn "import logging" code/` 返回 0；无 logger，靠 print/返回值传递状态 |
| `app.py` 单文件 8878 行 / 62 路由 | 严重 | 无服务层/领域层分离，路由里塞业务逻辑 |
| 依赖声明残缺 | 高 | `pyproject.toml` 只声明 fastapi/uvicorn，缺 pydantic/httpx/openai（实际用了 pydantic 但没声明） |
| 明文 API key 入库 | 高 | `configs/local/llm-gateway.local.json` 含 `sk-2112...`（虽在 .gitignore，但无 git 所以从未生效） |
| 40+ 脚手架端点 | 高 | 13 个 `llm-card-lifecycle-append-*` preflight/dry-run 端点包围一个从不执行的 append |
| JSON 文件持久化 | 中 | `JsonFileAsrLiveSessionRepository`，无 DB、无并发控制（仅原子写） |
| 测试方向偏移 | 中 | 测试名 `test_..._does_not_probe_audio` / `without_calling_llm`——验证"不做 X"而非"做对 X" |

### 3.2 前端：不是"丑"，是把 no-op 状态机渲染成了 UI

技术栈：**纯原生 HTML/CSS/JS**，`index.html` 252 行 + `styles.css` 1291 行 + `app.js` 3349 行，无任何框架/构建工具/类型系统/状态管理。`app.js` 用全局 `let currentSessionId=""` 等十几个全局变量管状态。

**真正的问题不是视觉，是信息架构**。`index.html` workbench 主列堆了约 15 个 section，其中 **8 个是流程状态面板**：

```
桌面准备度 / 桌面运行时边界 / 原生桥契约 / 原生运行时 /
ASR 交接 / 麦克风合同 / 真实麦克风准备度（Blocked）/ 成卡链路
```

每个面板默认显示"正在读取桌面 X 边界"——**把 internal no-op 脚手架状态当产品功能展示给用户**。真正的产品功能（转写、建议卡片、状态板）被埋在流程面板之间。"验收反馈"面板甚至让用户往 `<textarea>` 里贴原始 `Report JSON` / `Feedback JSON`——这是开发者调试 UI，不是产品。

工具栏 8 个按钮半数是内部黑话：`工件主线`、`闭环预览`、`Synthetic Preview`、`主线试运行`。普通用户无法理解。

**desktop_tauri**：5 个 Rust 源文件全 no-op（`lib.rs` 每个 `#[tauri::command]` 返回 `NoopBridgeResponse{captures_audio:false, spawns_process:false, safe_to_execute_real_action:false}`），但目录里有 **22 个 `.policy.json`** 描述这些命令"应该"做什么。前端 `app.js` 还专门维护 `desktopTauriNoopRunResultCollectorState`、`desktopMicAdapterNoopInvocationState` 等状态来渲染这些 no-op 结果——**前后端协同把空壳包装成了可见功能**。

### 3.3 ASR：真实可用，但被错误地当成"已验证不可用"

FunASR 和 sherpa-onnx **都真实接入了**：`transcribe_funasr.py:86-93` 真实 `AutoModel`+`generate()`，已下载 ModelScope iic 模型；`transcribe_sherpa_onnx.py:80-84` 真实 `OnlineRecognizer`，模型已就位。两者都产出了真实转写 JSON。`asr_bakeoff` 评测框架真实计算 CER/entity F1/latency。

**但 ASR 被误判为"不可用"**。7/4 真实麦克风测试 FunASR 输出"我是我样"/"六 六"——这个结果**糟糕到不可能是 FunASR 的真实水平**（FunASR 是成熟的中文 ASR）。更可能是：
1. 音频采集管线从未建好（全 no-op），那次测试的音频质量不可控——采样率/声道/格式/静音段可能全是错的；
2. 流式分块参数错误；
3. 模型未正确加载或热词未生效。

**也就是说：ASR 质量门禁（drv-032 `not_exited`）阻塞了整个项目，但这个"不可用"结论可能部分是音频管线 bug 伪装成的 ASR 质量问题。** 因为从未真正建过实时采集链路，所以从未在干净的真实音频上验证过 ASR。

## 4. 主链路为何一直没完成（根因排序）

### 根因 1（主因）：流程脚手架成瘾
项目把"为每个功能先建 plan→approval→readiness→preflight→evidence"当成默认工作方式。典型证据：
- DRV-043→044→045→046→047→048→049→050，**8 层脚手架**只为"批准一次本地 FunASR synthetic smoke 执行"，而该执行至今未批准。
- pcweb-064~076 连续 13 个 plan 都是"card lifecycle append"的不同 dry-run/preflight/disabled-run 变体，**没有一个真正创建了卡片**。
- desktop_tauri 22 个 policy.json vs 5 个 Rust 源文件。
- reset 文档第 5 节自己承认：`readiness→preflight→approval packet→no-op binding→dry-run→preview→wrapper→report` 这条链"对隐私安全有用，但没有改变决定性产品状态"。

### 根因 2：ASR 质量门禁变成阻塞门禁
`tools/asr_quality_decision_gate.py`（1007 行）组合了 product-value-batch、FunASR-readiness、模型下载审批、公开音频决策四道子门禁，条件苛刻到始终 `not_exited`，于是所有下游（真实麦克风 pilot、card 创建）全被阻塞。而门禁本身又成了源源不断的新工作（drv-033~050）。

### 根因 3：双轨代码从未合并
- **离线真实路径**（`asr_runtime/scripts/`）：transcribe→`meeting_analysis.call_llm()`（真实 HTTP）→state events→snapshot→markdown。LLM 真实可调，但要手动 CLI 编排。
- **在线模拟路径**（`web_mvp/backend/`）：ASR event 文件→EvidenceSpan→规则 candidate→**LLM disabled**→card not_created。

两条路径在 `call_llm()` 这个函数处断开，从未接通。web backend 的 471 个测试全跑在模拟路径上，真实路径几乎没有集成测试。

### 根因 4：安全顾虑压倒功能交付
18 个 `FALSE_SAFETY_FLAGS`、路径遍历防护、symlink 拦截、`configs/local` 阻断、`safe_to_*_now` 标志——安全意识优秀，但异化为"先写安全边界→再写审批包→再写就绪门→最后才考虑实现功能"，功能永远排在最后。`meeting_analysis.call_llm()` 写好了却不敢在 web backend 调用，就是典型。

### 根因 5：无工程基线
**零 git 提交**。所有工作在一个无版本控制的工作区里堆叠，没有 commit/branch/PR/CI，没有"可回溯的增量"。这违反最基础的生产级开发规范，也让"哪些代码是真的能跑的"变得不可回答——只能靠读 855KB decision-log 去猜。

## 5. 方向是否错了

| 维度 | 判断 | 依据 |
|---|---|---|
| 产品定位（中文技术会议实时 Copilot，证据驱动） | **对** | 差异化清晰，product-requirements 写得扎实 |
| Mac-first | **对** | 用户本机 Mac |
| 默认本地 ASR / 不默认付费 | 方向对，**执行受阻** | 真实语音召回 0.0，但可能是音频管线 bug 而非 ASR 本身 |
| 自建薄主线（不 fork） | 合理但**代价高** | ASR 未达标前自建桌面壳边际价值低 |
| **SDD+TDD 每步 plan-approval-evidence** | **错（过度）** | 异化为 wrapper 循环，reset 自己承认 |
| LLM 只做 request draft 不执行 | **过度保守** | 导致 LLM 环节从未真正验证 |
| 不下载模型/不下载公开音频 | **过度保守** | DRV-019 审批包长期卡住 ASR 验证，直到 7/4 才用缓存跑通，已晚 |
| 桌面端优先（vs 纯 Web） | **存疑** | 桌面端 17 天只做到 no-op shell；若先把 Web 主链路打通再包桌面，可能更快见到可用产品 |

**结论**：产品方向没错，技术选型基本对，**错的是方法学（过度规划/过度安全/脚手架优先）和优先级（先建桌面壳而非先打通主链路垂直切片）**。ASR 是否需要 pivot 到远程，应在"用干净真实音频重测"之后再决断，现在下结论太早。

## 6. 优化方案与计划

### 总原则

1. **停止生产新文档**。新工作只允许改 6 个决定性状态之一（ASR quality exit / real-mic readiness / 真实卡片 / 真实 LLM 调用 / 真实会议反馈），其余一律不算进展。
2. **垂直切片优先于水平铺开**。先让一条真实音频走完全链路，再扩功能，而不是先把每一层都建到"ready"。
3. **接通双轨**。把 `meeting_analysis.call_llm()` 接进 web backend，让 LLM 真实可调（带开关 + 用量记录）。
4. **建立工程基线**。立刻 git init + 首次提交 + 分支策略 + CI。
5. **删减脚手架**。40+ 个脚手架端点、22 个 policy.json、65 个 tools 里的 37 个 approval/readiness 脚本——归档而非删除，但不再投入。

### Phase 0：止血与基线（1–2 天）

- [ ] `git init` + 首次 commit 全量现状（含 .gitignore 已有）；建立 `main` + `dev` 分支；后续每个 Phase 一个 PR。
- [ ] 把 `configs/local/llm-gateway.local.json` 的明文 key 迁到 `.env`（python-dotenv），代码从环境变量读。
- [ ] 补 `pyproject.toml` 依赖：pydantic、httpx、python-dotenv、structlog。
- [ ] 引入结构化日志（structlog），至少覆盖 app.py 的请求入口、ASR 调用、LLM 调用、错误路径。
- [ ] 写一份 `DECISION.md`：冻结当前状态为 "Local Shadow Preview"，声明不再追加 readiness/wrapper 文档。
- [ ] 归档 `tools/` 中 37 个脚手架脚本到 `tools/_archive/`，从测试集移除对应的"不做 X"测试。

### Phase 1：ASR 路线决断（3–5 天，与 Phase 2 并行起跑）

**目标**：回答"本地 ASR 到底能不能用"，用数据决断而非门禁。

- [ ] **建一个最小可用真实音频采集**：Mac 上用 `sounddevice`/`pyaudio` 录 30 秒干净中文技术口播（非 ffmpeg avfoundation 的 ad-hoc 调用），保存 16kHz mono wav。这是 17 天里从没正经做过的事。
- [ ] **用干净音频重测 FunASR streaming + sherpa**：固定热词表，跑 `asr_bakeoff` 评测，记录 CER / 实体 recall / latency。
- [ ] **三个分支并评**：
  - A. 本地 FunASR（pinned 模型 + 热词 + 正确音频管线）
  - B. 本地 sherpa（快路径，中文实体增强）
  - C. 一个远程中文 ASR API（作为 pivot 对照，仅评测，不默认开启）
- [ ] **决断**：若 A/B 在干净真实音频上实体 recall ≥ 0.7，继续本地路线；若仍 < 0.5，pivot 到 C 并把"远程 ASR 默认关闭、用户显式开启"作为产品形态。
- [ ] **不要**再加新的 readiness/gate 文档。结论写进 `DECISION.md` 一段话即可。

### Phase 2：主链路垂直打通（1–2 周，核心）

**目标**：一条真实音频 → 实时转写 → 状态 → 真实 LLM → 真实卡片 → 纪要，端到端跑通。**只做这一条线，不铺新功能。**

- [ ] **后端分层重构 `app.py`**（8878 行 → 拆分）：
  - `routers/`（session、live_asr、report、llm、desktop）
  - `services/`（asr_service、state_service、suggestion_service、llm_service、report_service）
  - `repositories/`（保留现有 JSON repo，预留 DB 接口）
  - `schemas/`（pydantic models，复用 core 的 contracts）
  - 保留所有现有真实端点，**删除 40+ 脚手架端点**。
- [ ] **接通实时 ASR**：web backend 起一个 `/live/asr/stream` WebSocket，前端推音频 chunk → 后端调 FunASR/sherpa 流式推理 → 推 partial/final 事件到 SSE。复用 `streaming_contract.py` 的事件契约。
- [ ] **接通 LLM**：把 `meeting_analysis.call_llm_with_usage()` 提取成 `services/llm_service.py`，从 `.env` 读 base_url/key；suggestion_service 在 final segment + cooldown 后真实调用 LLM 生成结构化缺口建议；记录 provider/model/prompt_version/调用次数/token（满足 product-requirements 第 6 节）。
- [ ] **真实成卡**：`card_status` 从 `not_created` → `new`，写入 `MeetingStateEvent`，落 SSE。
- [ ] **会后纪要**：复用 `session_snapshot.build_markdown_report()`，接真实 LLM 生成带证据的 Markdown/JSON 导出。
- [ ] **一条 E2E 测试**：用一段预录真实音频跑完整链路，断言"产生了 ≥1 张真实卡片 + 卡片引用了 active EvidenceSpan + LLM 被真实调用且记录了用量"。这是 471 个测试里**缺失的那一个**。

### Phase 2.5：方案考量卡片——从"缺口雷达"到"会议助手"（3–5 天，Phase 2 跑通后）

**背景**：原产品定位是"工程缺口雷达"（product-requirements.md:104 "不替用户做架构裁决"、feature-map.md:73 "明确不做实时替用户做架构决策"），后端 5 条 gap 规则 + LLM prompt 写死只允许 `owner_gap/rollback_gap/test_verification_gap/metric_monitoring_gap`（`meeting_analysis.py:82`）。这使产品只回答"漏了什么"，不回答"正在讨论的方案怎么样"——差异化不足，容易退化成"带证据的待办清单"。

**目标**：在不违反"不替用户决策、证据驱动、低幻觉"原则的前提下，新增"方案考量卡片"——基于证据抛出**替代方案/考量点/追问**，语气与缺口卡一致（"是否考虑过 / 建议确认"），对象从"缺的项"扩展到"方案内容"。

**卡片类型族**（与现有 `*.gap` 并列，复用同一建议卡片管道）：

| 类型 ID | 语义 | 示例 |
|---|---|---|
| `approach.alternative` | 替代方案 | "灰度 5%→25%→全量，是否考虑过中间加 50% 档观察期？" |
| `approach.consideration` | 考量点 | "用降级开关做 rollback，是否明确过自动触发条件 vs 手动？" |
| `approach.precedent` | 先例引用（Phase 2+，需多会议记忆） | "此方案与 v2.1 灰度类似，当时 P99 延迟上升" |

**落地清单**：
- [ ] **后端**：`meeting_analysis.py` 的 `ALLOWED_CARD_TYPES` 增加 `approach.alternative/consideration`；`build_prompt()` 增加一段"可输出方案考量卡，但每张必须引用 evidence_span_id，且措辞为'是否考虑过/建议确认'，不得输出'应该/必须/不可'等裁判式结论"；`asr_live_events.py` 新增 `approach.*` 的 gap_rule_id 映射（沿用现有 `_suggestion_candidate_payload` 结构）。
- [ ] **风险门禁**（比缺口卡更严，因为方案建议幻觉风险更高）：
  - 每张考量卡**必须**引证据原文（无证据不输出，复用现有 gate）
  - 置信度门槛比缺口卡 +0.1（如缺口 ≥0.6，考量 ≥0.7）
  - 同议题同类考量 5 分钟冷却（复用现有 cooldown）
  - 单场会议考量卡数量上限默认 ≤3（缺口卡 ≤8），避免刷屏
  - 降级优先级：ASR 低质量/LLM 超时时，考量卡先于缺口卡沉默
- [ ] **前端**：新增 `.suggestion.approach` 变体（靛紫 #a78bfa 左边框，区别于缺口卡的状态色），灯泡图标，type 标签"方案考量 · 替代档位/增量回归"。已在 `设计稿/workbench-v1.html` 演示。
- [ ] **评测**：在 `asr_bakeoff` / product-value gate 增加"方案考量卡保留率"与"误报率"指标，与缺口卡分开统计；考量卡误报率门禁 ≤30%（比缺口卡 ≤20% 略宽，因主观性更强）。
- [ ] **Go/No-Go**：若真实会议中考量卡保留率 <25% 或误报率 >40%，停用该类型，回退为纯缺口雷达——不强行上方案建议。

**这是产品方向的关键增强，但不是回归"通用 AI 助手"**：考量卡仍受证据约束、仍非决策、仍可降级。它填补的是"方案质量提醒"空白，而非变成会议问答机器人。

## 7. 风险与决策点

| 决策点 | 选项 | 建议 |
|---|---|---|
| ASR 路线 | 本地 / 远程 / 混合 | Phase 1 用干净音频重测后定；**不要在重测前 pivot** |
| 前端栈 | 原生 / React / Vue | React+TS+Vite（Tauri 友好，生态大） |
| 桌面 vs Web 优先 | 先桌面 / 先 Web 主链路 | **先 Web 主链路打通**（Phase 2），桌面壳 Phase 4 再包；17 天证明先建桌面壳是低效的 |
| 脚手架代码 | 删 / 归档 | 归档到 `_archive/`，移出测试集，不再投入 |
| LLM 执行 | 保持 disabled / 带开关启用 | **带开关 + 用量记录启用**；安全靠开关和日志，不是靠永远不调用 |

**最大风险**：继续在本地零成本 ASR 路线上加 wrapper 而不重测，主链路仍不会完成。Phase 1 的干净音频重测是整个计划的命门——它要么解锁本地路线，要么给出 pivot 到远程的数据依据。两种结果都能让项目前进，唯独"不重测 + 继续加门禁"会继续原地打转。
